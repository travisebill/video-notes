## 一、主題與背景

FAI 6.3「AI Agents 代理人」是陳縕儂教授在 FAI 6.x Generative AI 章節的收尾之作，也是 FAI 整個 6.x 系列中最長的一講（38 分鐘）。影片從原模型「只能出一張嘴、不能真的做事」的痛點出發，介紹 Agent 的完整技術堆疊：Planning、Reasoning、Memory、Tool Use，並延伸到多 Agent 編曲、Human-in-the-Loop 安全設計，以及近期興起的 Harness Engineering 系統化工程概念。

影片開頭用一個生動案例點出 Agent 的必要性：使用者要求原模型訂 101 餐廳，原模型表面上答應得頭頭是道，看起來像專業客服，最後卻沒做任何事——沒訂到、沒做任何動作，只是「嘴上說說」。這正是 Agent 要解決的核心落差：讓模型具備 take action 的能力，可以呼叫 API、實際下單、執行指令，而不僅僅是產出文字描述。

從強化學習（Reinforcement Learning）的視角定義 Agent，是一個能感知環境（perception）、做出動作（action）、影響環境的實體（entity）。Rational Agent 會選擇能最大化效用函數（utility）的動作。當代語言 Agent 通常透過文字作為介面，與瀏覽器、作業系統、檔案系統、使用者等外部環境互動，核心仍是大型語言模型，外層則包覆感知、規劃、推理、記憶、工具呼叫等機制。

Agent 的四大核心能力依序展開。第一是 Planning：把高階目標拆解成可執行的小步驟。例如「去舊金山旅行」可拆成訂機票、訂飯店、安排行程，每個小任務再分配給對應的 Agent 或工具執行。第二是 Reasoning：Chain-of-Thought 思維鏈讓模型在回答前先寫下思考過程；ReAct 更進一步，要求模型先行動、觀察結果、再決定下一步，這對需要查詢或實作的任務特別關鍵——以 Apple Remote 為例，標準模型直接猜答案會錯，加入 search action 後正確率明顯提升。

第三是 Memory，分短期（short-term）與長期（long-term）。短期記憶放在 context window 內，但長度有限，任務複雜時容易遺忘開頭；長期記憶則寫入外部儲存，透過 retrieval 機制在需要時取回。開源框架 Letta 展示了 Agent 如何累積生活經驗、跨日保留記憶。第四是 Tool Use：Toolformer 論文證明模型可以學會在適當時機呼叫外部工具，例如計算機、QA 系統、維基百科 API——但前提是要有人工整理的訓練資料教模型正確的 API 呼叫格式，否則連 GPT-4 都會給出錯誤參數。

多 Agent 系統則引入 orchestrator 編曲概念，由一個總指揮 Agent 拆分任務、分派給多個專責 Agent 執行，如同交響樂團指揮。Human-in-the-Loop 是當前重要的安全設計：在關鍵決策點要求人類確認，避免錯誤自動化執行。Claude Code 偏向高度確認、安全優先；OpenHands 偏向自主執行、效率優先但風險較高。這中間的安全與便利權衡，是 Agent 落地最核心的實務問題。

Harness Engineering 一詞近期興起，指的是語言模型加上周邊的 context、工具、sandbox、orchestrator 等整套支撐系統——LM 本身可能不是最關鍵，harness 設計才是讓 Agent 在真實世界可靠運作的核心。

**核心議題一句話**：當代 AI Agent 是語言模型加上 Planning、Reasoning、Memory、Tool Use 四大能力的系統組合，搭配 Multi-agent 編曲與 Human-in-the-Loop 安全設計，並以 Harness Engineering 概念整合成可落地的工業級系統。

## 二、章節脈絡

### Section 1｜為何需要 Agent 與 Agent 定義（00:00 ~ 08:00）

**重點摘要：** 原模型只能「出一張嘴」，沒辦法真的做事；Agent 從 RL 視角定義為能感知環境、做出動作、影響環境的實體；Rational Agent 會選擇能最大化 utility 的動作；當代語言 Agent 透過文字介面與外部環境互動。

**內容：**
- 原模型痛點：只能說不能做
  - 範例：訂 101 餐廳，原模型表面上答應，最後沒做任何動作
- Agent 定義（RL 視角）：
  - 能感知環境（perception）
  - 做出動作（action）
  - 影響環境的實體（entity）
- Rational Agent：選擇能 maximize utility 的動作
- 互動模式：環境給刺激 → Agent 感知 → Agent 做 action → 環境改變
- 當代 Language Agent 透過文字作為介面
  - 環境：瀏覽器、OS、檔案系統、使用者
  - 核心仍是 LLM，外層包覆各種機制

> "Agent 裡面有一個非常重要的定義就是他可以 make action，對 action 就是他可以動作，比如說他可以去扣一個 API 他可以定位他可以幫你扛派有扣的執行扣的等等，而不是只是嘴上說說。"

### Section 2｜Planning 規劃（08:00 ~ 13:00）

**重點摘要：** Planning 把高階目標拆解成可執行的小步驟；例如「去舊金山旅行」可拆成訂機票、訂飯店；Web agent 範例展示如何把「找附近修車廠」拆解成多個瀏覽器動作；Multi-agent 系統引入 orchestrator 編曲概念。

**內容：**
- Planning 定義：把 high-level 目標拆解成一系列 actions
- 範例：去舊金山旅行
  - 高階目標：去舊金山旅行
  - 拆解：訂機票、訂飯店、安排行程
  - 每個子任務分配給對應的 Agent 或工具執行
- Web 動作範例：找附近修車廠
  - 輸入搜尋關鍵字 → 選擇位置 → Click search button → 查看 review
  - 整個過程是一連串 Web actions
- Multi-agent 系統：
  - 由 orchestrator（編曲者）拆分任務、分派給多個 Agent
  - Orchestrator 角色類似交響樂團指揮、team leader
  - 需要追蹤每個子任務的 progress、動態 revise plan

> "Planning 是什麼呢就是規劃嘛什麼叫規劃就是你得到一個 high level 的目標然後你把這個目標呢 Dec 可能沒辦法一次就到位...你把它拆解成旅行所以需要因為它在國外所以需要定機票需要定飯店。"

### Section 3｜Reasoning 推理（CoT + ReAct）（13:00 ~ 20:00）

**重點摘要：** Chain-of-Thought 讓模型在回答前先寫下思考過程；ReAct 要求模型先行動、觀察結果、再決定下一步；Apple Remote 案例展示標準模型直接猜答案會錯，加入 search action 後正確率明顯提升；Reasoning 與 Action 兩者缺一不可。

**內容：**
- CoT（Chain of Thought）：
  - 模型不是直接給答案，而是先寫下思考過程
  - 模擬人類的 Mental Process
  - 數學題範例：「5 顆球買了 2 罐每罐 3 顆最後有幾顆」→ 寫下推理過程而非直接給答案
- ReAct（Reasoning + Acting）：
  - 先做 action（搜尋、查詢）→ 觀察結果 → 決定下一步
  - Apple Remote 案例：
    - 標準 LLM：直接猜 iPod（古早東西）
    - + Reasoning：猜 iPhone/iPad/iPod Touch
    - + Action + Reasoning：先 search Apple Remote → 看結果 → 給出更準確答案
- 結論：Reasoning 與 Action 兩者缺一不可
  - 只思考不行動：會錯
  - 只行動不思考：會盲目亂找
- Reasoning 可擴展 action space：
  - Prompt 從只有 Observation 變成 Observation + Reasoning
  - 可處理更複雜任務

> "Standard 的這個 Lead 是他的問題好像就是什麼 Apple Remote 然後他叫他找說那些 Device 可以跟 Apple Remote Connect...如果我們讓他做 action 他先 SearchApple Remote 他就...找了一下 Apple Remote Search Apple Remote 在幹嘛...然後他就會覺得好他可以回答了。"

### Section 4｜Memory 短期與長期（20:00 ~ 26:00）

**重點摘要：** Short-term Memory 放在 context window 但長度有限；Long-term Memory 寫入外部儲存並透過 retrieval 機制取回；Letta 框架展示 Agent 如何累積生活經驗、跨日保留記憶；需要 retrieval model 在需要時才取用相關記憶。

**內容：**
- Short-term Memory：
  - 放在 context window 內
  - 任務複雜時容易遺忘開頭
- Long-term Memory：
  - 寫入外部儲存（如每日日記）
  - 透過 retrieval 機制在需要時取回
  - 類比：失憶患者每天結束前寫日記、隔天閱讀
- Retrieval Model：
  - 不相關的 memory 不放入 context，節省長度
  - 結合 relevance（相關性）與 recency（時效性）評分
- Letta 開源框架：
  - 模擬非常多生活瑣事存進 memory
  - 需要回答問題時 retrieval 相關 memory

> "短記憶是在 context window 內，但是長度有限制...所以我們就發現說有些東西感覺不能只在 Shorten Memory 你要不實地寫到你的 Lonton Memory 裡面。"

### Section 5｜Tool Use 工具呼叫（26:00 ~ 30:00）

**重點摘要：** Toolformer 論文證明模型可學會在適當時機呼叫外部工具（計算機、QA 系統、維基百科）；但需要人工整理的訓練資料教模型正確的 API 格式，否則連 GPT-4 都會給出錯誤參數。

**內容：**
- Toolformer 概念（2023 論文）：
  - 回答問題時有些計算不必自己做，交給外部工具
  - 計算機、QA 系統、Machine Translation、Wikipedia API
  - 將「查詢工具 → 取得答案 → 整合到回答」的流程 train 進模型
- 實際挑戰：API 呼叫並不容易
  - 即使是 GPT-4 也會犯錯（給出錯誤模型名稱、錯誤 API 格式）
  - 修法：人工整理訓練資料，示範正確的 API 呼叫
  - 教模型理解 API 規格、正確放置參數
- 落地：當代 Agent（Claude Code 等）能正確使用工具，是因為有 fine-tune 過的工具呼叫資料

> "他要說是文字轉成語音然後他要用 Horetop 裡面的這些東西 GP4 他們兩個其實都有問題 GP 他在這裡就是 Horetop 哪一個不存在的 Model。"

### Section 6｜Multi-agent、Human-in-the-Loop、Harness Engineering（30:00 ~ 38:26）

**重點摘要：** Multi-agent 由 orchestrator 拆分任務分派給專責 Agent；Human-in-the-Loop 在關鍵決策點要求人類確認；Claude Code 偏安全、OpenHands 偏自主執行；Harness Engineering 強調 LM 周邊支撐系統比模型本身更關鍵。

**內容：**
- Multi-agent 編曲：
  - Orchestrator 拆分任務、分派給專責 Agent
  - 透過 Language 當作最重要載具與環境互動
  - 產生文字描述 API 呼叫，再交給環境執行
- Human-in-the-Loop：
  - 在關鍵決策點要求人類確認
  - 提升 reliability，避免錯誤自動化
  - 範例：稅務表單填寫中途聯繫財務長確認
- 安全 vs 便利的權衡：
  - Claude Code：高度確認、安全優先
  - OpenHands：自主執行、效率優先但風險高
  - 選擇取決於使用情境
- Harness Engineering：
  - 近期興起概念
  - 強調 Agent = LM + 周邊支撐系統（context、tool、sandbox、orchestrator）
  - LM 本身可能不是最關鍵，harness 設計才是核心
- 演進脈絡：Prompt Engineering → Context Engineering → Harness Engineering

> "Horne's 大家現在就會覺得就有人說我們的 Agent 其實就像是 Model 加上旁邊這些有的美的 Horne's，那 Horne 是 Majju 的意思...你要能夠駕馭它你不需要給它好的這些 Majju 這樣就衝走了這樣。"

## 六、核心主旨總結

當代 AI Agent 是大型語言模型結合 Planning、Reasoning、Memory、Tool Use 四大能力的系統組合。Planning 將高階目標拆解為可執行步驟；Reasoning（CoT + ReAct）讓模型先思考再行動，特別是 ReAct 的「action → observation → next action」循環對需要查詢或實作的任務至關重要；Memory 區分短期（context window）與長期（外部儲存 + retrieval），讓 Agent 能跨 session 累積經驗；Tool Use 則讓模型學會在適當時機呼叫外部工具，但這需要人工整理的訓練資料教模型正確的 API 格式。當系統複雜度提升後，Multi-agent 編曲（orchestrator 拆分任務）成為主流架構，而 Human-in-the-Loop 則是當前最重要的安全設計。最後，Harness Engineering 概念提醒實作者：Agent 的成敗往往不在 LM 本身，而在於周邊的 context、工具、sandbox、orchestrator 等整套支撐系統的設計品質。
