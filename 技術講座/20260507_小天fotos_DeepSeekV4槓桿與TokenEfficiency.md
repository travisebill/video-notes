## 一、主題與背景

講者小天 fotos 為 AI 工程實踐者，長期關注模型部署與多 Agent 編排。本片的核心論點：**DeepSeek V4 已經帶來一個歷史性的槓桿窗口**——透過 V4 Pro（大模型顧問）+ V4 Flash（小模型執行者）的組合，以及極低的 API 成本（緩存命中每百萬 token 僅 2 分人民幣），讓「用白菜價做出高價值產品」成為現實。後半部深入分享他自行實踐的 **Harness 工程架構**——如何用多個小模型 Agent + 任務編排實現端到端全自動交付。

## 二、章節脈絡

### Section 1｜Anthropic 的顧問策略：大小模型組合的典範轉移（00:00 ~ 04:00）

**重點摘要：** Anthropic 提出的 OPS（顧問策略）驗證了「小模型執行 + 大模型只在需要時介入」的 Token Efficiency 模式。

**內容：**
- Anthropic 4/9 發布「顧問策略」：讓最便宜的模型幹活，最聰明的模型當顧問
- 只在執行者卡住時，顧問出手給 400–700 token 的指點
- HiQ + OPS 顧問在 BrowseComp 上分數從 19.7 → **41.2**（翻倍）
- 成本比 Opus 單獨跑低 **85%**

> 「前沿級別的推理能力只在執行者需要的時候才介入，其餘時間都維持執行者級別的成本。」

### Section 2｜Token Efficiency：AI 產業的真正方向（04:00 ~ 07:00）

**重點摘要：** Token Efficiency 已從敘事變成全行業共識——實現 AGI 更重要的不是造最強模型，而是讓小模型在大模型指導下勝任絕大多數任務。

**內容：**
- 產業現象：模型廠商普遍發布 1T + 300B 的大小模型組合（千問、螞蟻百靈、小米、DeepSeek）
- Token Efficiency 定義：小模型 + 大模型指導 + Harness 配合 → 極致成本節省 + 槓桿化
- 講者因千問 3.6 27B 的 SOTA 性能買了 RTX Pro 6000（6 萬多人民幣）
- 27B 模型已可成為 Token Efficiency 的「核心執行單元」

**個人實測：**
- NVFP4 精度本地部署，品質無損耗、更高並發
- 睡前給 6 個任務，Kimi K1.6 / GLM 1.1 指揮 Claude Code → Pipeline 半小時清空全部任務

> 「以後早上起來任務列表全部清空了，而我只是用了一些電費。你說這叫不叫槓桿？」

### Section 3｜DeepSeek V4 Flash：白菜價的超級模型（07:00 ~ 11:00）

**重點摘要：** DeepSeek V4 Flash 的核心突破——上下文壓縮、100 萬上下文、華為晶片——讓 API 成本幾乎歸零。

**內容：**

| 特性 | 影響 |
|------|------|
| **上下文壓縮** | 極省顯存，推理成本極低，降價空間大 |
| **100 萬上下文（1000K）** | 小任務不會觸發上下文壓縮，無性能損失 |
| **華為晶片** | 正式版將進一步降價 |
| **280B Flash 版 1000K 上下文** | 僅約 10GB 顯存 |

**價格：**
- 緩存命中：**每 100 萬 token = 2 分人民幣**（近乎免費）
- 緩存未命中：約 1 元/百萬 token
- Agent 任務中約 90% token 在緩存（輸入），10% 在輸出

> 「2 塊錢一個億的緩存，這是不是跟不要錢差不多了？下半年華為晶片大規模交付的話，價格要低於現在各種 coding plan。」

**100 萬上下文的真正意義：**
- 現有模型 200K–260K，稍微複雜任務就會滿 → 自動壓縮 → 性能下降、遺忘
- 1000K 窗口讓每個分解後的小任務永遠在窗口內完成

### Section 4｜兩級槓桿：從 API 到本地部署（11:00 ~ 13:00）

**重點摘要：** 講者定義的兩級槓桿架構。

**內容：**

| 級別 | 架構 | 成本 | 適合 |
|------|------|------|------|
| **第一級** | V4 Pro API（規劃）+ V4 Flash API（執行） | 極低（買 token） | 所有人 |
| **第二級** | V4 Pro API（規劃）+ 本地部署 Flash（執行） | 約十幾萬人民幣 | 最大化槓桿 |

> 「電費換生產力。」

### Section 5｜Harness Engineering：槓桿的真正關鍵（13:00 ~ 16:00）

**重點摘要：** Token Efficiency 不是有厲害的小模型就夠了——真正讓槓桿實現的是一整套 Harness（編排系統）。

**內容：**

講者實現的四層 Harness 架構：

**第一層：DAG 任務流程編排**
- 參考 Anthropic「解偶手和腦」的博客
- 用有向無環圖（DAG）描述任務流程
- 每個節點是獨立 Agent，運行在獨立沙盒
- 任務拆到極小 → 在一個上下文窗口內完成
- 節點間有預定義溝通方式（一對一、一對多）

**第二層：AI 迭代的編排系統**
- CLI 化編排構建系統
- Agent、Prompt 片段、資源都可複用組合
- 已跑通的編排成為 AuxTrader 的知識庫，越用經驗越豐富

**第三層：抽卡式任務解決**
- 一個 Task + 一個編排跑多次（多副本）
- 擇優錄取 / cherry pick
- 產生的數據用來優化編排 → Agent 節點加專屬 skill

**第四層：模型組合 AB 測試**
- 同一編排用不同模型組合運行
- 量化指標：工具調用成功率、失敗分布、token 成本、運行時間

> 「你需要的不是模型，而是軟體。這是目前 AI 世界裡沒有標準答案的地方——也是最有價值的一塊。」

### Section 6｜硬體平台選擇指南（16:00 ~ 18:30）

**重點摘要：** 根據三種角色給出具體的硬體/平台建議。

**內容：**

| 角色 | 優先選擇 | 理由 |
|------|----------|------|
| **普通開發者**（預算 ~3 萬） | Mac > 4090 48G > GB10 > AMD | Mac 可本地跑更大模型，一人夠用 |
| **創業者 / 一人公司**（技術驗證） | GB10 > 4090 48G | CUDA 平台必須，NVFP4 精度，可研究並發效率 |
| **老闆 / 產品化**（生產環境） | 多卡 CUDA（Pro 6000+） | 必須有經濟模型：token 怎麼轉化成錢 |

**軟體平台建議：**
- 非產品驗證：Lama.cpp / Ollama / LM Studio 都好用
- 做槓桿驗證（創業者）：**主攻 VLLM 和 SGLang**，必須 Linux 平台

> 「如果你想吃 AI 行業的飯，CUDA 設備遠勝於 Apple Silicon。」

**講者自白：** 本人在等 M5 → 背叛蘋果 → 買了 RTX Pro 6000

### Section 7｜三個行動建議（18:30 ~ 結束）

**重點摘要：** 給所有人的三條實戰建議。

**內容：**

1. **盡可能用小模型** — 別整天只訂閱 Claude Max。改環境變數把 Sonnet/Haiku 設為 V4 Flash，啟動 `--model opspline` 模式（計畫用 Pro，執行用 Flash）
2. **建立評估體系** — 量化！用 API 大模型跑基線，再用小模型對比，知道什麼任務用什麼模型最划算
3. **把業務流程原子化** — 不用自己手搓完整 Harness（會有很多現成的出來）。把業務流程拆成節點，串行/並行關係想清楚，每個流程單元確保在 100 萬 context 內完成

> 「你現在唯一要做的不是追最強的模型，而是充分理解 Token Efficiency，用好小模型，設計好編排。」

## 三、關鍵概念定義

| 概念 | 定義 |
|------|------|
| **Token Efficiency** | 讓小模型在大模型指導下勝任大多數任務，以極低成本實現槓桿化的 AI 產業方向 |
| **OPS（顧問策略）** | Anthropic 提出：便宜模型執行 + 聰明模型僅在卡關時介入 400–700 token |
| **Harness Engineering** | 多 Agent 任務編排工程：分工、規劃、流程控制、無人值守、高並發、端到端交付 |
| **DAG（有向無環圖）** | 描述多節點任務流程的資料結構，每個節點是獨立 Agent |
| **抽卡式任務解決** | 同一任務跑多個副本，擇優錄取 + 產生優化數據 |
| **Context Compression** | 上下文超過視窗限制時自動壓縮 → 導致性能下降和遺忘 |
| **NVFP4** | NVIDIA 4-bit 浮點精度格式，Blackwell 架構支援，品質損耗低但顯存需求大降 |
