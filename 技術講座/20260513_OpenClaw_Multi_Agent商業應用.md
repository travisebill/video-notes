## 二、章節脈絡

### Section 1｜開場：為什麼代理會是系統的核心（00:00 ~ 04:00）

**重點摘要：** 講者提出核心論點——編碼代理不僅是開發工具，而是未來軟體系統的核心建構模塊。展示從工具到系統的思維轉變。

**內容：**

- 系統提示（System Prompt）定義代理能做什麼、使用哪些工具
- > 「編碼代理正在、也將成為你軟體系統的核心建構模塊。我在這上面下注，很多人也在這上面下注。」

### Section 2｜OpenClaw 技術堆疊：從 Pi 到完整平台（04:00 ~ 09:00）

**重點摘要：** 講者拆解 OpenClaw 的技術架構——底層 Pi 框架、中層代理核心（Agent Core）、上層 OpenClaw 平台外掛機制——展示模組化設計如何讓不同場景共用同一核心。

**內容：**
- **Pi**：Python 最小化代理框架
  - Session 支援完善——建立 session、代理、串流所有資訊回傳
- **Agent Core**：代理的核心邏輯層
- **PiI**：統一的 LLM 抽象層
- **終端 UI 介面**
- **OpenClaw 外掛機制**：
  - 多通路路由（Multi-Channel Routing）
  - 提供者協作（Provider Orchestration）
  - SAP 代理、Gateway 支援
- > 「OpenClaw 建立自己的外掛機制是因為它有不同使用場景和不同需求。但它基於 Pi 的核心機制，並充分利用了核心。」

### Section 3｜真實商業案例：銷售流程的代理自動化（09:00 ~ 15:00）

**重點摘要：** 全場最核心的段落。講者展示一個真實的 B2B 銷售自動化系統：一個客戶專屬一個代理、監控信箱、分析需求、查詢 CRM/ERP、生成回覆草稿——全程由多代理協作完成。

**內容：**
- 使用場景：公司收到詢價郵件（Requests for Proposals）→ 需要自動處理
- 系統架構（從頭設計的全新思維）：
  1. **郵件進入** → 監控信箱
  2. **Gateway 路由** → 根據郵件內容決定發送給哪個代理
  3. **多代理架構**：每個客戶一個專屬代理
     - **Agent MD**：代理的通用配置——理解自己在系統中的角色、如何反應
     - **Customer MD**：客戶專屬配置——特定條款、折扣、存取權限
  4. **Session 重用**：每個案例（Case）建立或重用既有 session → 代理記得之前的對話脈絡
  5. **工具層**：CLI 工具連接 CRM 和 ERP 系統 → 代理能查詢客戶資料、訂單歷史、庫存等
  6. **產出**：在信箱中生成草稿郵件 → 使用者可以直接編輯和發送
- 安全設計：
  - CLI 包裝資料存取——代理擅長使用 CLI
  - 自有沙箱保護資料安全
  - Nvidia 的 OpenClaw 安全策略（Open Shell）是值得關注的方向

### Section 4｜系統 Demo 與幕後運作（15:00 ~ 18:00）

**重點摘要：** 現場展示管理儀表板和代理在後台的實際運作——多個工具呼叫、迴圈迭代、最終產出草稿的全過程。

**內容：**
- 儀表板 Demo：
  - 信箱監控介面
  - 郵件關聯到 Case（每個 Case 就是一個代理 Session）
  - 代理呼叫多個工具（查 CRM、查 ERP），在迴圈中迭代
  - 最終產出：草稿郵件出現在使用者信箱中
- > 「對使用者來說，體驗就是：我看信箱，有一封新郵件，關聯到一個案例，然後出現一封草稿——我可以自由編輯。但在幕後，所有這些代理都在協作。」

## 三、關鍵概念定義表

| 概念 | 定義 | 出處／應用 |
|------|------|-----------|
| Pi（Python 代理框架） | OpenClaw 底層的最小化代理框架，支援 Session、Agent、Streaming | OpenClaw 技術堆疊的基礎 |
| Agent MD / Customer MD | 代理的配置文件——Agent MD 定義角色和行為，Customer MD 定義客戶專屬規則 | 多代理系統中的配置分層 |
| Session 重用 | 每個案例（Case）對應一個 Session，代理能記住之前的對話歷史 | 代理保持上下文的核心機制 |
| CLI 工具包裝 | 將 CRM/ERP 資料存取包裝為 CLI 工具，讓代理透過命令列介面安全存取 | 代理存取企業系統的設計模式 |
| Gateway 路由 | 根據郵件內容決定將任務發送給哪個客戶專屬代理 | 多代理系統的入口層 |
| 沙箱安全 | 隔離代理的執行環境，確保資料安全 | Nvidia Open Shell 是值得關注的方向 |
| 草稿生成模式 | 代理生成回覆草稿，由人類審核後發送——「代理輔助、人類決策」 | 銷售自動化的 UX 設計 |

## 四、核心主旨

> 編碼代理不只是幫你寫程式碼的工具——它們是你下一個軟體系統的核心建構模塊。從郵件路由到 CRM 查詢、從客戶意圖理解到回覆草稿生成——一個「每客戶一代理」的多代理架構，加上 CLI 工具層和 Session 重用，讓代理能夠像真正的團隊成員一樣協作。底層的 Pi 框架極簡到可以拆開重組，現在就是開始實驗的最佳時機。
