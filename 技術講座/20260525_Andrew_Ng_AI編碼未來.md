## 核心主題：軟體工程與 AI 編碼的未來

### 軟體如樂高積木

Karpathy 在一年前的 AI Dev 大会就分享過這個比喻：他將軟體工具、框架、API 視為**樂高積木**。

- 只有一種白色積木 → 能建一些東西，但不有趣
- 混合黑色、藍色、紅色、黃色積木 → 能建更複雜的東西
- 積木種類越多，組合方式呈**指數成長**

過去一年，AI 編碼代理的崛起讓所有人都能更有效地快速組合這些積木。

**兩類積木都在爆炸性增長：**
- **AI 積木**：大語言模型、RAG、Agent 工作流等
- **非 AI 積木**：UI 元件、資料庫持久化、身分驗證系統等

> 「因為 AI 編碼，這些積木正以我們從未見過的速度激增。每天都有越來越多的積木，不論 AI 或非 AI，讓我們使用。這太棒了，特別是當 AI 編碼讓我們能讓 AI 代理來組裝這些積木時。」

---

### AI 編碼的滲透率：80% vs 100%

Karpathy 觀察到一個關鍵問題：團隊應該使用多少 AI 編碼？

他觀察到一個悖論：

| 團隊類型 | AI 編碼滲透率 | 實際產出 |
|----------|--------------|----------|
| 大多數團隊 | 80% AI + 20% 人類審查 | 人類審查瓶頸，浪費大量時間 |
| 前沿團隊 | 100% AI | 速度極快，大量產出 |

> 「我自己幾乎 100% 都是 AI 編碼。我想分享為什麼我認為這最後的差距影響很大。」

**Karpathy 的觀察：**
- 80% AI 編碼看起來不錯，但人類審查那 20% 會耗費大量時間
- 100% AI 編碼的團隊在短時間內能完成多得多的事
- 這不只是寫 code 的問題，**review code 也適用**

> 「如果你試圖自己 review code，你就成了瓶頸，而且效果也不好。」

**重要澄清**：這不是宗教。NASA 寫 50 行控制太空船的程式碼當然應該手寫。但 Karpathy 看到的趨勢是：**前沿團隊正在走向 100% 或接近 100% AI 編碼**。

### 第二層效應：PM 瓶頸

AI 編碼帶來了一個**令人著迷的第二層效應**。

Karpathy 在去年 7 月的 newsletter 中寫過這個觀察：

傳統流程：```idea → prototype → software engineering → users → feedback → PM/product management → refine```

AI 編碼讓軟體工程快了 10-100 倍。這意味著**決定要建什麼（PM 工作）成了新的瓶頸**，而不是實際的建構。

**一個奇怪的趨勢開始出現：**

| 傳統比例 | 新興比例 |
|----------|----------|
| 1 PM : 8 engineers | 1 PM : 2 engineers（很奇怪）|
| | 1 PM : 1 engineer（更奇怪）|

Karpathy 觀察到的趨勢：與其保持 1 engineer + 1 PM，不如把這兩個人**合併成一個會coding的PM 或會做PM的工程師**。兩者都會 coding 的團隊移動速度極快，速度本身就是巨大的競爭優勢。

---

### 各種瓶頸：設計、法務、行銷

當軟體工程速度提升 10-100 倍，其他一切都變慢了。

| 瓶頸類型 | 說明 |
|----------|------|
| **設計瓶頸** | 設計完→實作，甚至設計師直接寫 code（跳過 Figma）更快 |
| **法務合規瓶頸** | 花 3 個月寫 code，法務花 1 個月審核 → 可接受。但花 1 天寫 code，等 1 個月法務 → 災難 |
| **行銷瓶頸** | 工程團隊出貨速度太快，行銷來不及告訴大家 |

Karpathy 觀察到的結論：越來越多的**小型跨功能團隊**（包含工程師，但也有其他人）移動很快。一個好消息是：一個會有效 prompt AI 的工程師，即使不是專業行銷人，也能做基礎的行銷工作。AI 可以幫助補足這些跨領域的技能差距。

---

### AI 原生團隊的組成

不是所有事情都能由一個 10 倍團隊完成。但當你有許多 AI 原生團隊，各自只有有限的溝通和清晰的 API 邊界，這種結構也能 Scale。

> 「當軟體工程速度快了 10-100 倍，其他一切都看起來很慢。」

### Job Apocalypse 的真相

Karpathy 觀察到矽谷圈子裡流行討論「AI 工作末日論」——AI 會搶走所有工作，大家都要上街頭。

**他的實際觀察：我完全沒看到這個發生。**

- 商業媒體比大眾媒體更準確，因為商業媒體靠精準吃飯
- 過去一兩個月，商業媒體已經在修正這個敘事：就業末日被推遲了、削減人力不是普遍現象、聯準會調查顯示少有公司因為 AI 裁員

Karpathy 承認：
- 許多人感受到各階層級的工作不安全
- 疫情後過度招聘和零利率時代的招聘導致很多人被裁
- 就業形勢並不完美
- **但 AI 工作末日論在一些圈子裡只是暫停，並非消失**

> 「這就是為什麼我熱衷於持續投資讓我們所有人變得更好。我只是沒看到這個（工作末日）發生。」

## Context Hub 工具介紹

### 問題：知識截止日期導致的幻覺

積木（building blocks）增長太快，導致編碼代理的知識截止日期（可能是一年前或幾個月前）**經常產生幻覺或使用已棄用的 API**。

> 「舉例來說，如果你今天讓 coding agent 呼叫 OpenAI API，它會使用 chat completions API——那個更舊的、已棄用的、呼叫更舊模型的 API。因為網路上大多數訓練資料用的就是這個舊 API，即使新 API 已經出來很久了。」

---

### 解決方案：即時文件拉取

**Context Hub** 是 Karpathy 和同事 Rakhit Prasad、Singhyal Hote 一起打造的工具，目的是**給 AI agent 最新、最即時的文件**。

功能：
- 執行一次搜尋，回傳套件列表
- 透過 Chubb 拿到 `open.ai/chat`，Fetch 最新 OpenAI 文件
- 知道 GPT-5.5、知道 Responses API

使用方式：
- 適合與 Cloud Code 搭配使用
- 生成一個六行、Mounted 檔案（有點長，人類可以讀）
- 人類也可以閱讀它來獲取 context並自行呼叫 API

> 「我禱告很多人，你們可能都經歷過我的 coding agent 對 OpenAI、Gemini、各種資料庫服務產生幻覺 API 呼叫。餵入最新資訊真的非常有幫助。」

## CodeDream 發布

Karpathy 在演講中宣布了一個新項目：**CodeDream**（現場 demo，Wi-Fi 不太穩定）。

### 這不是一堂課，是一次對話

CodeDream 的定位：
- **不是一堂課**，是一次對話
- 目標：展示如何透過 coding agent + Context Hub，用 OptoJ 的 NPR 文件來寫 code
- 想像成一個價值數十億美元的開發環境，可以隨時打斷提問
- CodeDream 創建了一個語音 agent，試圖像主持人一样回答問題

### 教學環境：練習使用現代編碼代理

CodeDream 的 build environment 是一個瀏覽器內的終端機，可以：

1. 從投影片複製指令（如 `npm install`）
2. 貼上執行
3. 啟動 open code（開源 coding agent）
4. 開始用 Python 或其他語言寫 code

> 「這是一個練習環境，讓你練習使用現代 coding agent，結合 Context Hub 等工具。」

## Karpathy 的 AI 工程師技能模型

當被問到招聘 AI 工程師時，Karpathy 看重三項能力：

| 技能 | 說明 |
|------|------|
| **有效使用 coding agents** | Cloud Code、Gemini CLI、Open Codex 等 |
| **了解豐富的積木世界** | 挑戰是這些積木太多、變化太快 |
| **通用技能** | 基礎產品管理、其他領域能力，能快速組裝東西 |

**技能組合的關鍵：**
- 知道有哪些積木可以組合
- 知道要建什麼

> 「我看到對這些技能的組合有大量未被滿足的需求。即使我的團隊，也找不到足夠多這樣的人。」

## 總結

Karpathy 的核心訊息：

1. **軟體工程正在加速**：AI 編碼讓 80% 的工作自動化，但那 20% 的人類審查會形成瓶頸。前沿團隊走向 100% AI 編碼來獲得真正的高速。

2. **PM 是新的瓶頸**：軟體工程快了 10-100 倍，決定要建什麼比實際建構更重要。跨功能小團隊（會coding的PM）移動最快。

3. **更多瓶頸相繼出現**：設計、法務、行銷都落後了。AI 原生團隊用 AI 輔助補足這些差距。

4. **Job Apocalypse 未來不會來**：商業媒體已經在修正這個說法。AI 工程師的需求仍然巨大。

5. **Context Hub + CodeDream**：兩項工具，一個解決 AI agent 文件過時問題，一個提供練習現代 AI 編碼的環境。

6. **人類需要的互補技能**：隨著 coding agents 越來越強大，人類需要培養與 AI 協作的技能——知道有哪些積木、了解如何組裝、以及最重要地，知道要建什麼。
