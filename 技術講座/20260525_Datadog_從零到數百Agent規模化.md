## 已建置的三個 Agent

### 1. BITS AI SRE Agent

- 自動化 AI SRE，能自動調查系統問題
- 採用現成模型打造聰明的 DevOps/SRE 類型 Agent
- 背景：現在用 AI 編碼產生大量程式碼，但很難掌握整個程式碼庫的所有內容，出問題時需要自動調查

### 2. BITS AI Dev Agent

- 根據錯誤和問題自動寫 code、部署修復
- 結合「理解系統」和「修復系統」兩種能力

### 3. Security Analyst Agent

- 接收可疑訊號，自動調查系統和人員行為
- 類似 SRE Agent 的概念，但專注於安全事件
- 節省大量原本需要人工執行的時間

> 「coding 已經被 Agent 解決了——也許還沒完全解決，但我們很接近了。SRE 也正在朝這個方向發展。」

## 規模化的六個關鍵教訓

### 一、Code/Agent First 的 UX

**核心問題：** UX 設計師只考慮人類用戶，忽略了下一批主要用戶是自動化 Agent。

**應該做的事：**

| 行動 | 說明 |
|------|------|
| **新增 LLM.txt** | 類似當年 robots.txt 的做法，幫助 Agent 理解你的網站 |
| **支援 .md 文件** | 讓文件可以被機器讀取 |
| **採用 MCP（Model Context Protocol）** | 讓功能可被 Agent 呼叫 |
| **API 優先** | 每個功能都應該有 API 介面 |
| **定期用自己的 Agent 測試** | 不要只用人類操作，假设顧客會用 Agent 操作 |

> 「這就像是新的貝佐斯 API 宣言。所有團隊都應該用 Agent 友好的介面來處理每一項功能。」

**實務建議：** 走過一遍團隊所有 UI 功能，檢查哪些無法用非瀏覽器或電腦操作的 Agent 存取。

---

### 二、積極主動而非被動回應

**核心問題：** 大多數 Agent 只在聊天時被觸發，但應該是事件驅動（event-driven）。

**兩種模式對比：**

| 被動模式（聊天觸發） | 主動模式（事件驅動） |
|-------------------|-------------------|
| 人類輸入問題 | 系統自動觸發 |
| 即時回應 | 背景持續運行 |
| 只能處理少量任務 | 可同時管理大量任務 |

**原則：**
- 除了開發者日常用的 coding agent，其他都應該在背景某處運行
- 不要每天早上對每個 Agent 逐一指示，應該讓它們有既定任務並主動執行
- 使用 **Temporal** 等工具確保 Agent 的持久性和自我修復能力
- 適當地沙箱化（sandbox）每個 Agent，防止資料外洩

> 「聊天只是一種互動形式。聊天不應該是你觸發所有 Agent 的主要方式。它們大多數時候應該是事件驅動的。」

### 三、Eval（評估）

**這是演講中最強調的一點。**

**Datadog 犯過的錯誤：** 一開始沒有仔細思考如何評估 Agent，導致不斷投入開發時間修修改改，卻不知道是否有實質進步。

**評估的三個步驟：**

```
Offline Eval → Online Eval → Living Eval（持續更新）
```

| 階段 | 做法 |
|------|------|
| **Offline Eval** | 建立小型但具代表性的資料集，可衡量、可重複運行 |
| **Online Eval** | 收集真實運行數據（觀測資料、點擊等），確認在生產環境中的表現 |
| **Living Eval** | 持續用真實資料更新 offline 資料集，因為客戶使用方式會改變 |

**實際操作：**
- 觀察 Live 互動和 traces（追蹤）：所有工具呼叫、人類輸入、使用的 context
- 每筆 trace 都應該能夠被註解（標記好/壞、哪些工具應該被使用）
- 追蹤錯誤叢集（clusters of errors），理解整體表現
- 最終可以用 Agent 來評估其他 Agent，形成「誰來監督看守者」的遞迴

> 「我們有這些 Agent 自動改善系統的功能，看著它們自動提升表現非常酷。」

### 四、Multiplayer 協作

**不只是多人在同一螢幕上操作（上一代的多人協作），而是 Agent 之間的协作。**

三種 multiplayer 形式：

| 形式 | 說明 |
|------|------|
| **人類 ↔ 人類** | 仍存在，但重點變成「能看到隊友的 Agent 在做什麼」 |
| **人類 ↔ Agent** | annotation（註解）、corrections（修正）、解釋為什麼某件事發生 |
| **Agent ↔ Agent** | 在安全的封閉網路內，讓 Agent 之間互相溝通和觸發 |

**實際做法：**
- 建立團隊或組織專用的 **Skills Hub** 或 **MCP Hub**
- 能夠看到、分享、改編其他人的 Agent
- 使用 EKS cluster 限制網路存取，確保安全
- 類似 RPA 的新做法：共享螢幕讓 Agent 看見該怎麼做，或 Agent 分享它做的事並解釋

### 五、框架與模型無關性

**關鍵認知：**
- 框架採用率在過去一年翻倍成長
- 不管用哪個框架（OpenAI Agents、Anthropic、LangGraph、Pydantic 等），都建議使用現成框架避免重複造輪
- 多數組織已經是多模型（multi-model）配置——模型會不斷互相超越，今天最好的模型明天可能就被替換

**建議：**
- 保持模型無關（model agnostic）
- 保持一定程度的框架無關
- 不要由上而下規定團隊用什麼框架或模型，這會限制可能性和學習
- 自動化系統可在每次新模型發布時自動執行評估

### 六、Agent 的Bitter Lesson

> 「真正能快速利用新現成模型的一般性方法，最終會獲勝。模型會不斷變得更好。你為 Agent 做的每一個小調整，最終可能都需要拆除。」

**原則：**
- 確保大而聰明的模型有正確的工具（tools）和技能（skills）來完成你想要的事
- 坦然接受「做完工作後要拆除」的狀況
- **記憶（Memory）非常關鍵**：保持模型無關但累積的記憶和 context，即使新模型出現也能保留過去的學習

**Memory Agent 的作用：**
- 從觀測資料、traces 中提取語意知識
- 提取與未來相關的 context
- 修剪過時的 context

## 預測：AI 的未來走向

| 預測 | 說明 |
|------|------|
| **在工作崗位上學習（Learning on the job）** | 現在的 Agent 普遍是「建構一次就不斷運行」，不會從錯誤中學習。RL across enterprise 將成為焦點。建議：現在就做好資料日誌，不要遺失 |
| **自我改善的 Agent** | 從人類觀察中學習，Synthetic environments（合成環境）將崛起——用模型模擬你的產品和人類互動方式 |
| **更長時間運行的 Agent** | 從 30 分鐘→數小時→數天，持久性 Agent（durable agents）越來越重要 |
| **Agent 環境下的 ACL** | 經典的存取控制概念如何應用在 Agent 場景？Agent 可以代表你行動，但不應該有你所有的權限 |
| **多模態** | Computer use 幾乎來了——但目前速度慢、會點錯地方，還需一兩年 |
| **文字 → 雙眼和耳朵** | 語音和即時互動正在回歸，整合各种碎片 |
| **生成式 UI** | 標準案例有精心設計的 UI，但也會看到大量即時生成、自定義的 UI（如自訂儀表板、按鈕） |

## 總結

從一兩個 Agent 擴展到數百個 Agent 的六個關鍵：

| 關鍵 | 行動 |
|------|------|
| **Code/Agent First UX** | 確保所有功能可被 Agent 存取，建立 MCP、使用 API |
| **Proactive over Reactive** | 讓 Agent 在背景事件驅動運行，不要只用聊天觸發 |
| **Eval** | 建立 offline/online/living 評估迴圈，用真實資料持續改善 |
| **Multiplayer** | 建立 Skills Hub，支援人類↔Agent、Agent↔Agent 的協作 |
| **Framework & Model Agnostic** | 保持靈活，自動化評估新模型，不要由上而下限制 |
| **Bitter Lesson** | 信任大模型 + 工具，坦然拆除過時的自訂邏輯，記憶是保留學習的關鍵 |
