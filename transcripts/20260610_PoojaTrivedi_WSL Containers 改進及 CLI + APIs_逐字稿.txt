## 00:07:09 — Demo 3：GPU 加速 AI 容器

> 現在你已經看到怎麼把 Linux 容器內的服務 expose 到 Windows 主機。我想展示更有趣的事：這些容器能怎麼用上 Windows 主機上的 GPU，無論是 AI 場景或圖形應用都行。

Pooja：
> 我已經把容器 build 好並預熱了，所以它已經在跑（為了省時間）。
>
> 它是 Jupyter Notebook web server，一樣聽 port 8888 並用 port mapping expose 出來。
>
> 我開始跑這些 cell，同時跟你說這個酷 AI 容器做什麼事：
> 它會拉下 GPT-2 模型，然後在 Windows 主機的 GPU 上 fine-tune，然後做 torch compile 產生 **fused Triton 核心**。這些 Triton 核心是最佳化過的，讓模型在 GPU 上跑超快。
>
> 接下來跑個 demo：compiled 模型 vs uncompiled（eager mode）對比。你可以看到 compiled 模式快非常多。
>
> 這一切發生時，Windows 主機的 GPU 已經被借用，在 Linux 容器內完成所有運算。

> 交回給 Craig 之前，讓我快速秀一下我啟動這個 AI 容器的指令：

> `wslc run -it -p 8888:8888 -v /host/hf:/container/hf --gpus all myai:latest`
>
> - `-it`：互動模式
> - `-p 8888:8888`：port mapping
> - `-v`：掛載 Hugging Face cache 從 Windows 主機到容器內
> - `--gpus all`：讓容器用 Windows 主機的 GPU

Craig：
> 回到投影片，可以看到下一步。這個 demo 的重點是：你有兩種方式跟 WSL Container 互動。

## 00:09:44 — 為誰而做？三大客群

> 「為什麼我們要做這個？這是為誰做的？Microsoft 對這項技術的目標是什麼？」這分成三種客戶輪廓。

### 1. Container Developers
> 如果你是個容器開發者，這就是為你而做。如果你想 build 容器，我們想讓「上手使用」更簡單，並且整個體驗在 Windows 上流暢跑起來。
>
> 我們也想對自己的方向有意見：我們自己做了 API 跟 CLI，所以會做更自訂的指令把它們綁在一起。這就是為什麼我們 build 自己的 Linux container CLI，而不是用 Docker / Podman / Rancher Desktop 這些標準工具。
>
> 不過這些工具也都很好用，如果它們解決你的問題，你應該用。它只是另一種方法。這個 work 本身是 fully open source，VM 底層的改進也會回流到其他 container 技術。例如我們改進 cross-OS 檔案效能，Docker Desktop / Rancher Desktop / Podman Desktop 都會享受到。

### 2. App Builders / ISVs
> 我們也想為 App Builders 跟 ISV 做這件事。如果你在雲端或本地有一堆很棒的 Linux code，你想直接放在 Windows 應用程式內跑，我們想讓你用強大且標準的 API 在 Windows 應用裡做到這件事。

### 3. Enterprise IT Admins
> 最後，我們也為企業 IT 管理員做這件事。有些人想：在我的企業內跑 OpenClaw，想用 Linux container 化的方式跑，但要用我熟悉的 Windows 工具（像 Intune、MDE）管理它。
>
> 我們 build 這件事正是為這種場景。我想用這些強大的工具跟應用，並在 Windows 上跑 Linux，又不用擔心組織內有個看不見的 VM。
>
> 我想確保你拿到強大的隔離，又不用擔心你的資安工具打不穿這條界線。

## 00:16:42 — 架構說明

Pooja：
> 我相信你接下來最燒腦的問題是「這到底是怎麼運作的？」讓我解釋一下架構。

> 工作原理是：每個想要建立 Linux 容器的 Windows 應用程式，都會在背景被指派一個專屬的輕量 utility VM。所有該應用程式的 Linux 容器都共置在這個 VM 上。每個應用程式以及 CLI 流程都會拿到獨立的 VM。

> 這給我們：**hypervisor boundary per app**，加上資源隔離，避免衝突。應用程式的容器名稱、volume 名稱、網路都按 VM 做隔離，不會有衝突。

> 更細一點：左邊是 Windows OS，右邊是那個 VM（由 Windows app 透過 WSL Containers API 或 CLI 流程建立）。

> 進入 WSL Containers 世界的兩個入口：
> 1. **API**：透過 `wslcsdk` 這個 library（NuGet 套件）發布
> 2. **CLI**：`wslc` 或 `container` 執行檔

> 這兩個都透過 IPC 跟 WSL service 通訊。如果你用過 WSL，WSL service 已經存在於今天的世界。它的 job 是管理 WSL 跑的 distros、WSL VM 的 lifecycle、VM 跟 host 的通訊。
>
> 我們擴充了同樣的 service，現在也管理 VM 內跑的容器，也管理容器跟 Windows 主機上使用者之間的雙向資訊轉發。

> 在 Linux VM 內部，我們跑一個 **container runtime**。今天用的是 **Moby** — 也就是 Docker 用的開源引擎。
>
> WSL service 用 **HV socket** 跟 container runtime 通訊：開啟一條 HV socket 進到 Linux VM。它不是 network socket，是個專門的 socket，專門用於 VM 跟 hypervisor 之間的通訊。它是 Winsock API 的特殊延伸。
>
> 每個 VM 有點像一條點對點的專屬通道到 hypervisor — 每個 VM 都有獨立的通訊 channel。

> 一旦使用者發出 container 建立的命令，container runtime 就建立那些容器，WSL service 負責雙向轉發資訊。

## 00:19:47 — 儲存管理

> 接下來想快速講一下 storage。

Pooja：
> 每個 VM 內的 container runtime 用一個中央目錄存 container 相關資訊 — metadata、scratch space 等。通常是 `/var/lib` 跟 container storage 位置。
>
> 我們現在把這個位置 map 到 Windows 主機上一個虛擬磁碟（VHD）。
>
> **每個應用程式都拿到自己的 VHD**。CLI 流程也拿到自己的 VHD。
>
> 所以在 storage 層面，每個應用程式也是完全隔離的。

> 最後快速展示一下，當上面那個 `wslc run -it` 指令跑起來時，底下發生什麼事：
>
> - Volume mount：我要把 Windows 上 `C:\data` map 到 Linux 容器內的 `/data`
> - 我要建立 `debian:latest` 容器
>
> 這個指令發出去時，container runtime 建立 `debian:latest` 容器，並掛載 `/data` volume。
>
> 然後在底層，我們把 `/data` folder（或 volume）map 到 Windows 的 `C:\data` folder，用的是 **VirtIO FS**。
>
> VirtIO FS 是個相對新的 file sharing protocol，用於 VM 跟 hypervisor 之間的檔案分享。對我們來說，這裡有額外的複雜度：要翻譯兩種完全不同的 OS 之間的 file system 語意。
>
> 如果你用過 WSL，它今天用的是 **Plan 9**。VirtIO FS 在這個環境下大約快 2 倍。我們會持續投資這個領域，看各種替代方案讓它更快。
