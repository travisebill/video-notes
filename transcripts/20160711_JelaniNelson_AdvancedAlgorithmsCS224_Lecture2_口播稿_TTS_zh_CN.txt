Harvard CS224 高阶演算法第二堂课的主题是 Fusion Trees，焦点放在 word-RAM 模型下的静态前驱问题如何用位元层级的技巧逼近理论下界。整堂课的核心讯息是：当你面对资料量太大、无法放进单一 word 的情境时，要靠位元层级的平行处理与乘法技巧同时操作多个键，这也让 word-RAM 模型从纯比较模型走向真正的计算模型。

Section 1｜开场修正
课程开头，教授先修正上一堂提到的 Van Emde Boas 树空间错处。教授引用 CLRS 课本习题二十之一 Part F，承认该论证其实有个微妙的错误：用 hash table 换掉 array 并不能真正拿到线性空间，实际的空间复杂度会略差一些，原因是 hash function 的期望负载分析并不如预期。这个错误会放进第一次作业，让学生找出哪句话错并修正。Y-Fast Trie 的部分则维持原案，确认用间接层后确实得到线性空间，因为代表元素的设计让 hash table 的每个桶期望装根号 n 个元素。教授也顺道澄清，这次修正是针对论证本身的错误，并非否定 vEB 树本身的理论价值，vEB 树依然是时间最佳的选择。

Section 2｜前驱问题定义与历史脉络
接著教授重新定义前驱问题，分为静态与动态两种设定。静态设定下集合不变，动态设定则需要支援插入与删除。Fusion Tree 是 Fredman 与 Willard 在 1993 年提出的静态资料结构，查询时间达到对数 w 为底 n 的对数，比 vEB 树更快且更接近 word-RAM 模型的理论下界。后续 Anderson 与 Thorup 将其做成动态版本，更新时间是对数 w 为底 n 加上对数对数 n，属于确定性演算法。Raman 更进一步把更新时间压到对数 w 为底 n，属于随机演算法的期望值。本堂课只展示静态版本，前处理时间可以是多项式等级，这对静态应用来说是合理的代价，因为资料一进来就不变了。

Section 3｜为何需要位元层级技巧
接下来教授解释 Fusion Tree 的关键设计：让内部节点装 B = w 的五分之一次方个键，而树深度就是对数 w 为底 n 的对数。但问题在于每个节点必须只用常数时间处理，否则节点数的额外因子会把对数基底的好处抵消掉，回归到二元搜寻树的对数 n。更糟的是 K 个键每个是 w 位元，加起来超过一个 word，根本放不进单一暂存器，更别说要在常数时间内处理。为了解决这个问题，教授为你列出四个核心工具：乘法、草图压缩、word-level 平行处理、最高位元查找副程式。后者在 word-RAM 下用二分搜寻可以做到对数 w 时间，更进一步能做到常数时间，这是整个演算法能跑的关键。

Section 4｜单节点搜寻的位元构造
详细的单节点搜寻靠一个关键引理，你会看到：给定 K 个 w 位元数字，存在一个魔法常数使得这些数字乘上该常数后，重要位元会变得连续且没有空隙，可以直接用遮罩过滤出来。这个引理留给学生自己证明。一旦拿到这个常数，就能用乘法和位移把原本分散的重要位元压缩成连续的 1、0、1、1 这类 pattern，整个过程常数时间。接著用 word-level 平行处理把这个 pattern 跟预先准备好的遮罩向量比对，就能找出第一个非零的丛集是第几个。最后再做一次平行处理找出最高位元，决定要走哪个子树。整个节点搜寻只用常数个 word 操作，这就是 Fusion Tree 名字的由来，也是 word-RAM 演算法设计的原型。

Section 5｜结论与下周预告
单节点搜寻讲完后，教授为你强调 Fusion Tree 的核心思想：当资料结构放不进一个 word 时，就用位元层级的平行处理把多个键同时处理。这样的设计哲学将成为后续 word-RAM 演算法的设计原型，包含融合节点的各种变体、word-RAM 优先伫列等。教授预告下周正式进入 hashing 主题，会涵盖负载平衡、K-wise 独立性、字典问题、完美杂凑、线性探测等议题，为 cuckoo hashing 与 Bloom filters 铺路。word-RAM 主题告一段落后，下个阶段将进入更贴近实务应用的 hashing 工具箱。
