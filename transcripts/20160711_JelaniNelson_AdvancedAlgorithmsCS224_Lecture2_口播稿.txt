Harvard CS224 高階演算法第二堂課的主題是 Fusion Trees，焦點放在 word-RAM 模型下的靜態前驅問題如何用位元層級的技巧逼近理論下界。整堂課的核心訊息是：當你面對資料量太大、無法放進單一 word 的情境時，要靠位元層級的平行處理與乘法技巧同時操作多個鍵，這也讓 word-RAM 模型從純比較模型走向真正的計算模型。

Section 1｜開場修正
課程開頭，教授先修正上一堂提到的 Van Emde Boas 樹空間錯處。教授引用 CLRS 課本習題二十之一 Part F，承認該論證其實有個微妙的錯誤：用 hash table 換掉 array 並不能真正拿到線性空間，實際的空間複雜度會略差一些，原因是 hash function 的期望負載分析並不如預期。這個錯誤會放進第一次作業，讓學生找出哪句話錯並修正。Y-Fast Trie 的部分則維持原案，確認用間接層後確實得到線性空間，因為代表元素的設計讓 hash table 的每個桶期望裝根號 n 個元素。教授也順道澄清，這次修正是針對論證本身的錯誤，並非否定 vEB 樹本身的理論價值，vEB 樹依然是時間最佳的選擇。

Section 2｜前驅問題定義與歷史脈絡
接著教授重新定義前驅問題，分為靜態與動態兩種設定。靜態設定下集合不變，動態設定則需要支援插入與刪除。Fusion Tree 是 Fredman 與 Willard 在 1993 年提出的靜態資料結構，查詢時間達到對數 w 為底 n 的對數，比 vEB 樹更快且更接近 word-RAM 模型的理論下界。後續 Anderson 與 Thorup 將其做成動態版本，更新時間是對數 w 為底 n 加上對數對數 n，屬於確定性演算法。Raman 更進一步把更新時間壓到對數 w 為底 n，屬於隨機演算法的期望值。本堂課只展示靜態版本，前處理時間可以是多項式等級，這對靜態應用來說是合理的代價，因為資料一進來就不變了。

Section 3｜為何需要位元層級技巧
接下來教授解釋 Fusion Tree 的關鍵設計：讓內部節點裝 B = w 的五分之一次方個鍵，而樹深度就是對數 w 為底 n 的對數。但問題在於每個節點必須只用常數時間處理，否則節點數的額外因子會把對數基底的好處抵消掉，回歸到二元搜尋樹的對數 n。更糟的是 K 個鍵每個是 w 位元，加起來超過一個 word，根本放不進單一暫存器，更別說要在常數時間內處理。為了解決這個問題，教授為你列出四個核心工具：乘法、草圖壓縮、word-level 平行處理、最高位元查找副程式。後者在 word-RAM 下用二分搜尋可以做到對數 w 時間，更進一步能做到常數時間，這是整個演算法能跑的關鍵。

Section 4｜單節點搜尋的位元構造
詳細的單節點搜尋靠一個關鍵引理，你會看到：給定 K 個 w 位元數字，存在一個魔法常數使得這些數字乘上該常數後，重要位元會變得連續且沒有空隙，可以直接用遮罩過濾出來。這個引理留給學生自己證明。一旦拿到這個常數，就能用乘法和位移把原本分散的重要位元壓縮成連續的 1、0、1、1 這類 pattern，整個過程常數時間。接著用 word-level 平行處理把這個 pattern 跟預先準備好的遮罩向量比對，就能找出第一個非零的叢集是第幾個。最後再做一次平行處理找出最高位元，決定要走哪個子樹。整個節點搜尋只用常數個 word 操作，這就是 Fusion Tree 名字的由來，也是 word-RAM 演算法設計的原型。

Section 5｜結論與下週預告
單節點搜尋講完後，教授為你強調 Fusion Tree 的核心思想：當資料結構放不進一個 word 時，就用位元層級的平行處理把多個鍵同時處理。這樣的設計哲學將成為後續 word-RAM 演算法的設計原型，包含融合節點的各種變體、word-RAM 優先佇列等。教授預告下週正式進入 hashing 主題，會涵蓋負載平衡、K-wise 獨立性、字典問題、完美雜湊、線性探測等議題，為 cuckoo hashing 與 Bloom filters 鋪路。word-RAM 主題告一段落後，下個階段將進入更貼近實務應用的 hashing 工具箱。
