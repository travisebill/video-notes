這段節目是 FAI 第 3 章的尾聲，處理深度學習中的最佳化問題——給定 Loss Function 與模型結構後，要如何實際把參數 θ 推向最小值。林軒田教授把焦點放在三個關鍵的工程變量：起步位置、步伐大小、與每一步看到的樣本數。

第一個問題是「滾球會卡在局部低谷」。影片把 Gradient Descent 比喻為沿著 Loss 地形圖滾下的小球，從某個起點開始，計算當下位置的切線斜率，若斜率為負就往 θ 增加的方向滾一步；若斜率為正就往 θ 減少的方向滾。最終小球會停在斜率為零的位置，也就是局部低谷。然而深度學習的 Loss 地形遠比線性模型來得崎嶇——層數一多，局部低谷的數量呈指數級增加；從不同起點出發，最終停下的位置也會截然不同。這就是為什麼深度學習對「初始化點」極為敏感。實務上的解法是在多個隨機起點上各跑一次，最終選 Loss 最低的那一個 θ 作為候選解。

第二個問題是「步伐大小」。學習率（Learning Rate）若設得太大，參數可能直接跨過低谷，反而讓 Loss 上升；若太小，雖然方向正確，卻會在低谷附近卡住，動彈不得。理想的策略是「遠離低谷時大步走，接近低谷時小步走」——也就是自適應（adaptive）的學習率調整。現代最佳化器如 Adam、RMSProp 內建了這類機制，自動根據梯度歷史調整步伐；若使用固定學習率，則需要手動監控訓練曲線、逐步下調。

第三個問題是「每一步要看到多少資料」。標準的 Full-Batch Gradient Descent 每次看完所有訓練樣本才更新一次，方向最精準但速度最慢，尤其當資料量達到上萬筆以上時，每一次更新都要運算整體平均，效率極差；SGD（Stochastic Gradient Descent）則每看完一筆就更新一次，更新次數多，但單筆的梯度估計雜訊大；實務上的最佳折衷是 Mini-Batch SGD——一次取 B 筆（例如 32、64、128）平均後再更新，兼顧速度與穩定性，並且能充分利用矩陣與矩陣相乘的硬體加速效益。

影片最後提醒，深度學習的最佳化並不是「找到一個確定最佳的 θ」，而是「盡量少卡在糟糕的局部低谷」。Batch Size 通常建議固定，避免每次重新調整 cache 機制；每個 epoch 內對訓練資料做隨機洗牌，確保模型學不到資料順序的偽相關；Learning Rate 則建議參考已發表論文的設定，並結合 gradient clipping 等技巧處理梯度爆炸問題。

整段節目的核心訊息是：深度學習的最佳化既包含梯度下降的數學原理，也疊加了 Learning Rate、Batch Size、隨機化與 adaptive optimizer 等多項工程技巧；這些「小撇步」看似瑣碎，卻是把理論模型推向實際可用的關鍵所在。
